How should the SmartPOS experience be structured internally?

All three options use the same routes (/offer → /financing → /pre-qualified → /pos-demo). The question is how the SmartPOS phases are managed inside /offer.

/offer → 3 separate sub-routes
/offer/estimate
/offer/confirm
/offer/handoff
Each screen = own URL
Back button works

A — Sub-route wizard

Each SmartPOS phase gets its own nested route. Clean browser history, shareable deep links, back button works naturally.

Pros

  • Native browser navigation
  • Deep-linkable
  • Predictable

Cons

  • More routes to manage
  • State passing between routes
/offer → single component, internal states
URL stays: /offer
state: ESTIMATE
state: MICRO_CONFIRM
state: REFINED
state: HANDOFF
All state lives in React

B — Single-route state machine Recommended

SmartPOS is one component with 4 internal states. URL stays /offer throughout. Matches existing POSDemo pattern. Smooth animated transitions.

Pros

  • One route, zero URL noise
  • Smooth transitions
  • Matches POSDemo architecture
  • Simple state pass to /financing

Cons

  • Back button needs custom handling
/offer → 2 phases, 2 files
SmartPOS.jsx (estimate + confirm)
Handoff.jsx (specialist state)
Split at /offer and /handoff
Clean component boundaries

C — Two-file split

SmartPOS handles estimate + micro-confirm. Handoff gets its own component at /handoff. Clear separation between "discovery" and "transition."

Pros

  • Clean file responsibility
  • Handoff screen is standalone

Cons

  • Extra route for an intermediate state
  • More state passing hops
